# 红酒扫码内容体验项目需求调研报告

- 文档版本：v1.0
- 文档状态：调研稿
- 编写日期：2026-04-09
- 适用阶段：项目立项 / 方案确认 / MVP 范围定义

## 1. 项目概述

### 1.1 背景

本项目面向酒类消费后的品牌内容体验场景。用户在线下购买红酒后，品牌方为每件商品提供一个二维码；用户扫码后进入数字化内容页，查看该款红酒的品牌介绍、产区信息、风味说明，并触发一段音乐内容或歌单体验，从而提升产品的仪式感、差异化和品牌记忆点。

项目当前需要解决的核心问题包括：

1. 该项目是否应优先采用微信小程序，还是直接做网站/H5。
2. 行业内类似项目通常采用什么产品形态。
3. “一次性二维码”在技术和业务上如何更合理地定义。
4. 首个可落地版本应包含哪些必要功能，哪些内容适合后置。

### 1.2 目标

本次调研的目标不是生成详细设计稿，而是为产品立项和技术选型提供依据，形成以下输出：

1. 行业同类产品与公开案例判断。
2. 小程序与网站路线的适配性分析。
3. 一次性二维码能力的实现边界与业务建议。
4. 可执行的需求范围、后台能力和风险清单。

## 2. 调研结论摘要

基于截至 2026 年 4 月 9 日可查的公开资料、微信生态能力和酒类行业常见做法，本项目的结论如下：

1. `主入口建议采用微信小程序，辅以 H5 兜底页`。如果目标用户主要在中国大陆消费场景中通过微信扫码进入内容，小程序更符合用户路径、品牌触达和后续运营要求；但为了兼容微信外扫码、外部链接传播、海外访客和内容预览，仍建议保留 H5 页面。
2. `一次性二维码不应简单理解为“扫一次后永久作废”`。更合理的业务定义是“首次扫码激活并绑定，后续重复扫码按规则展示不同状态”。如果直接做成完全失效码，售后争议、误扫、信号异常、店员预扫等问题会显著增加。
3. `中国酒类/快消一物一码项目普遍偏向微信生态`，重点是防伪、溯源、会员、活动和复购；`葡萄酒国际案例更偏向网站/H5 内容页`，重点是酒标信息、品牌故事、营养/合规标签和互动内容。
4. `音乐内容是体验亮点，但也是合规风险点`。如果直接在项目内播放商业歌曲，需要明确版权授权；更稳妥的方式是跳转官方音乐平台歌单，或使用品牌自制/已授权音频。
5. `MVP 应以“扫码识别 + 酒款内容页 + 音乐页 + 码状态管理 + 后台配置”作为最小闭环`，先验证体验价值和扫率，再决定是否扩展到会员、积分、抽奖、复购商城或防窜货系统。

## 3. 调研范围与方法

本次调研主要采用以下三类资料：

1. 微信官方能力文档，用于判断小程序码、普通二维码拉起小程序、参数透传等实现边界。
2. 品牌官网、SaaS 平台方案页、公开案例页面，用于判断行业产品形态和交互方式。
3. 公开媒体与服务商案例文章，用于辅助判断酒类行业的主流落地路径。

需要说明的是，部分行业案例来自公开报道或平台展示页，能够反映`行业采用何种产品形态`，但不能替代对具体项目后台架构和运营数据的内部审计。

## 4. 业务场景理解

### 4.1 核心用户

1. 消费者：购买红酒后扫码，获取内容体验。
2. 品牌方运营人员：维护酒款文案、图片、音乐配置和活动规则。
3. 渠道/经销商：参与铺货、礼赠、终端展示或线下导购。
4. 客服/售后人员：处理无码、异常码、被占用、真伪咨询等问题。
5. 后台管理员：批量生成二维码、管理码状态、查看扫码数据。

### 4.2 核心场景

1. 用户购买后，使用微信扫码进入品牌内容页。
2. 用户查看该酒款的产区、品种、风味、饮用建议和品牌故事。
3. 用户点击播放对应音乐、氛围音频或歌单。
4. 系统识别该二维码是否首次激活、重复扫码、异常或失效。
5. 品牌方在后台替换酒款内容或音乐配置，无需重新印刷二维码。

### 4.3 项目本质

从业务性质看，这不是单纯的“产品介绍页”，也不是完整的“电商商城”，而是一个介于`品牌内容体验`与`一物一码数字化入口`之间的轻运营产品。其价值主要来自：

1. 提升用户对酒款的感知质量和品牌仪式感。
2. 将线下售出的实物商品变成可追踪的数字入口。
3. 为后续会员、复购、社交传播和活动运营留下接口。

## 5. 同类产品与项目调研

### 5.1 行业形态判断

从公开资料看，相关项目可分为三类：

1. `微信生态型`：以微信小程序、公众号、视频号为主入口，常见于中国酒类、快消和零售项目。
2. `网页内容型`：扫描二维码后进入 H5/落地页，常见于葡萄酒品牌故事页、营养标签页、海外酒庄内容页。
3. `一物一码平台型`：由 SaaS 平台提供二维码、后台、数据和模板，小程序或网页只是具体打开方式。

### 5.2 典型案例对比

| 案例 | 公开产品形态 | 主要能力 | 对本项目的启发 |
| --- | --- | --- | --- |
| 酒小二 | 微信小程序 | 酒类零售、用户触达、线上转化 | 中国酒类消费在微信内完成路径较短，小程序适合作为主入口 |
| 郎酒一物一码案例 | 小程序 + 一物一码体系 | 商品码、物流与终端数据打通、消费者触达 | 本项目如果后续做会员或真伪验证，可以沿这一路径扩展 |
| 茅台葡萄酒防伪溯源 | 微信扫码查询页 | 首次查询、非首次查询、无效码识别 | “首次/重复/无效”状态管理很适合直接借鉴 |
| 草料专属小程序 | SaaS + 小程序 | 用企业自有小程序打开二维码、内容管理、数据看板 | 说明市场已有“二维码 + 小程序品牌展示”的成熟需求 |
| Vinofyed | 网页平台 | 酒类 QR 营销、个性化落地页、分析后台 | 葡萄酒行业对 QR 内容页和多路径落地已有成熟需求 |
| Winetraqr | 网页平台 | 扫码查看产品、标签、品牌与供应链信息 | 葡萄酒包装与酒标内容展示更适合结构化网页呈现 |
| Frontera x Spotify | 品牌活动页 / 音乐平台联动 | 扫码后进入酒与音乐搭配内容 | “酒 + 音乐”在消费者沟通层面成立，具备可传播性 |
| Cupcake “Love is Wine” | Web 活动页 | 预生成有效码、扫码验证、互动体验 | 可参考其预生成二维码和状态校验思路 |
| Living Wine Labels | 原生 App / AR | 扫酒标进入增强现实故事体验 | 体验上限高，但开发与运营成本高，不适合当前 MVP |

### 5.3 归纳判断

这些案例反映出几个稳定趋势：

1. 在中国市场，只要入口依赖“微信扫码”，项目最终大多会向小程序靠拢。
2. 在葡萄酒行业，“扫码看内容”本身是成立需求，且用户愿意接受较强的品牌叙事和沉浸式信息呈现。
3. “扫码后听歌”属于品牌内容创新，不是行业标配，因此可以形成差异化，但前提是音乐资源合法可用、交互足够顺畅。
4. 防伪/真伪校验和品牌内容并不冲突，很多项目会把两者放在同一个扫码入口中。

## 6. 微信小程序与网站方案对比

### 6.1 对比维度

| 维度 | 微信小程序 | 网站 / H5 |
| --- | --- | --- |
| 微信内扫码体验 | 强，链路最短 | 一般，需要打开浏览器或内嵌 WebView |
| 品牌感与信任感 | 强，展示小程序主体与品牌名 | 中等，依赖域名与页面设计 |
| 微信生态运营 | 强，可结合公众号、服务通知、视频号 | 弱，无法天然承接微信生态能力 |
| 外部平台访问 | 一般 | 强，任意浏览器可访问 |
| 海外用户兼容 | 弱 | 强 |
| 内容富媒体承载 | 中等偏强 | 强 |
| 搜索引擎收录 | 弱 | 强 |
| 审核与合规门槛 | 较高，需要过微信审核 | 相对可控，但仍需内容合规 |
| 开发维护成本 | 前后端 + 微信接入 | 前后端相对简单 |
| 扫码后数据沉淀 | 强，便于后续微信内触达 | 中等，取决于登录与埋点体系 |

### 6.2 结论

如果本项目的主要交易场景发生在中国大陆，且消费者大概率使用微信扫一扫进入内容页，那么单做网站不是最优方案。网站虽然更通用，但会牺牲微信内链路和后续运营空间。

综合判断，本项目建议采用以下架构：

1. `主体验使用微信小程序`。
2. `二维码本体优先使用普通 URL 方案`，在微信内通过“扫普通链接二维码打开小程序”能力拉起小程序。
3. `微信外访问落到 H5`，作为兼容和分享页。

这样可以用一套二维码兼容两套访问场景，避免做两套完全独立的码体系。

## 7. 一次性二维码需求判断

### 7.1 官方能力边界

根据微信开放能力，小程序提供的是`小程序码生成能力`、`URL Link / URL Scheme`、以及`普通链接二维码打开小程序`等入口能力；但微信并没有提供现成的“二维码扫码一次后自动永久失效”的业务组件。

因此，所谓一次性二维码，本质上需要由业务后台自行完成：

1. 为每瓶酒或每份商品生成唯一 token。
2. 将 token 映射到某个酒款、订单或批次记录。
3. 用户扫码进入后由服务端校验 token 状态。
4. 首次扫码时激活并记录；重复扫码时按规则处理。

### 7.2 不建议做成“扫一次即彻底作废”

如果把码设计成“任何人扫过一次就直接废掉”，会出现以下问题：

1. 店员、导购、仓储人员可能提前误扫。
2. 用户首次扫码时因网络异常未看到内容，却已被判定为失效。
3. 用户更换手机或再次查看时无法进入体验页。
4. 外盒码被拍照转发后，消费者难以判断自己是否为首次合法使用者。

### 7.3 建议的一次性业务定义

建议将“一次性二维码”定义为`首次激活绑定型二维码`，而非绝对作废型二维码。推荐状态机如下：

1. `未激活`：允许首次扫码并创建会话。
2. `已激活-本人访问`：允许继续查看内容。
3. `已激活-他人访问`：显示“该码已被激活”，可选择仅展示只读内容或提示联系客服。
4. `无效码`：系统中不存在该码。
5. `失效码`：品牌方手动停用，或超出有效期。

### 7.4 包装与防复制建议

如果项目未来带有权益、会员或抽奖价值，仅靠公开二维码无法真正防复制，建议至少结合以下措施中的一项：

1. 将二维码放在包装内页、封签内侧或刮涂层下。
2. 对高价值产品使用“外码导流 + 内码激活”双码结构。
3. 在后台保留异常频次、异常地区、短时间多设备扫描预警。

## 8. 产品定位与目标建议

### 8.1 产品定位

本项目建议定位为：

`面向酒类消费后的品牌内容体验小程序，以一物一码为入口，为用户提供酒款介绍、音乐陪伴和品牌记忆，并为后续会员运营和真伪服务预留扩展空间。`

### 8.2 阶段目标

#### 第一阶段目标

1. 证明扫码后内容体验对用户有吸引力。
2. 证明“酒款内容 + 音乐”能提升停留和品牌感知。
3. 验证二维码管理、状态校验和后台配置流程是否可跑通。

#### 第二阶段目标

1. 引入会员绑定、礼赠关系或复购转化。
2. 引入积分、权益、限定内容或活动玩法。
3. 对接更严格的防伪、溯源、防窜货系统。

## 9. 功能需求建议

### 9.1 C 端用户功能

#### 9.1.1 扫码进入

1. 用户扫码后自动识别访问环境。
2. 微信内优先打开小程序。
3. 微信外打开 H5 兜底页，必要时提示“请在微信中打开以获得完整体验”。

#### 9.1.2 年龄确认与饮酒提示

1. 首次进入时进行 18+ 年龄确认。
2. 展示“理性饮酒”“未成年人禁止饮酒”等必要提示。

#### 9.1.3 酒款内容页

建议至少包含以下内容模块：

1. 酒款主视觉。
2. 品牌/酒庄介绍。
3. 产区与葡萄品种。
4. 年份、酒精度、容量、等级等基础参数。
5. 风味描述与口感关键词。
6. 醒酒建议、饮用温度、餐酒搭配。
7. 包装故事或品牌叙事内容。
8. 真伪状态或激活状态提示。

#### 9.1.4 音乐页

1. 每款酒可绑定一首歌、一段音频或一个歌单。
2. 用户可播放、暂停、切换。
3. 页面可展示“这款酒为什么搭配这段音乐”的说明。
4. 若采用第三方音乐平台，则跳转或唤起官方内容页。

#### 9.1.5 码状态反馈

1. 首次扫码成功时给出明确反馈。
2. 重复扫码时说明该码已激活。
3. 无效码或异常码时引导联系客服。

#### 9.1.6 分享与收藏

1. 支持转发酒款内容页。
2. 支持收藏品牌主页或小程序。
3. 可选地提供“送礼分享卡片”或“推荐给朋友”的轻分享能力。

### 9.2 B 端后台功能

#### 9.2.1 酒款内容管理

1. 新增/编辑/下线酒款。
2. 配置图文、参数、品牌故事、饮用建议。
3. 配置音乐资源、歌单链接或音频说明。

#### 9.2.2 二维码与码池管理

1. 按 SKU、批次或订单批量生成二维码。
2. 支持导入第三方印刷码或外部码池。
3. 查看码状态：未激活、已激活、失效、异常。
4. 支持单码查询、批量失效、批次统计。

#### 9.2.3 扫码记录与分析

1. 首扫人数与次数。
2. 复扫人数与次数。
3. 地区、时间、设备分布。
4. 酒款访问量、停留时长、音乐播放率。
5. 异常码或高频扫描预警。

#### 9.2.4 客服与申诉处理

1. 被占用码申诉。
2. 码不存在/无法识别反馈。
3. 真伪咨询入口。
4. 人工改绑或人工复核能力。

### 9.3 运营能力

1. 支持活动文案临时替换。
2. 支持节日主题皮肤或限时内容。
3. 支持针对不同渠道、地区或活动批次展示不同落地内容。
4. 支持后续接入优惠券、会员注册、留资和复购链接。

## 10. 非功能需求建议

### 10.1 性能

1. 首屏内容应尽量在 2 至 3 秒内可见。
2. 音频资源需要控制大小，避免首开卡顿。
3. 二维码校验接口应保持低延迟，减少扫码等待感。

### 10.2 内容配置灵活性

1. 内容字段应可配置，避免每新增酒款都要发版。
2. 音乐与图文资源建议走后台 CMS 管理。
3. 二维码和酒款内容应解耦，避免因改版重印所有物料。

### 10.3 安全

1. token 不应暴露可推导的业务含义。
2. 接口需具备防刷、限流和日志审计能力。
3. 后台需要角色权限控制。

### 10.4 可运营性

1. 扫码后的关键漏斗要有埋点。
2. 内容更新不应依赖开发重发版本。
3. 异常码和客服流程要能在后台定位到具体批次与酒款。

## 11. 合规与风险评估

### 11.1 酒类内容合规

根据《中华人民共和国广告法》的酒类广告相关约束，项目内容需要避免以下风险：

1. 诱导、怂恿饮酒或过量饮酒。
2. 出现饮酒动作或过度渲染饮酒场景。
3. 对未成年人形成不当吸引。
4. 使用夸大、功效化、疗效化表述。

因此，在小程序与 H5 中都建议保留年龄提示与理性饮酒文案。

### 11.2 音乐版权风险

若系统内直接播放商业音乐，需确认是否已获得信息网络传播相关授权。对当前项目而言，更稳妥的策略是：

1. 使用品牌自制音频或已购买授权音频。
2. 跳转到合法音乐平台上的官方歌曲/歌单页面。
3. 在后台明确记录每个音频资源的授权来源与有效期。

### 11.3 隐私与数据合规

如果项目收集手机号、位置、OpenID、用户画像或扫码轨迹，应补齐：

1. 隐私政策。
2. 用户授权提示。
3. 数据存储、权限与留存规则。

### 11.4 渠道与售后风险

1. 导购预扫会干扰“首次扫码”判断。
2. 包装外露码容易被复制传播。
3. 用户对“码被占用”的容忍度较低，需预设人工处理流程。

## 12. 技术路线建议

### 12.1 推荐方案

建议采用`小程序主体验 + H5 兜底页 + 后台码状态服务`的组合方案。

建议流程如下：

1. 后台生成唯一 token，并与酒款/SKU/批次建立映射。
2. 二维码优先使用普通 URL 形式，便于微信内外统一分发。
3. 微信内扫码时拉起小程序，并将 token 传入。
4. 小程序向后台校验 token 状态并获取酒款内容。
5. 后台返回不同状态和可访问范围。

### 12.2 为什么不建议只做小程序码

如果完全依赖小程序码：

1. 微信外扫码兼容性较差。
2. 海外用户和第三方社媒传播受限。
3. 物料统一管理和内容复用空间较小。

### 12.3 为什么不建议只做网站

如果完全依赖 H5：

1. 微信内品牌触达感会弱一些。
2. 无法充分利用小程序作为品牌资产沉淀入口。
3. 后续如果要加会员、服务通知或其他微信能力，迁移成本会上升。

## 13. MVP 范围建议

### 13.1 建议纳入 MVP 的内容

1. 扫码识别与环境路由。
2. 年龄确认页。
3. 酒款详情页。
4. 音乐播放页或歌单页。
5. 一次性激活与码状态判定。
6. 后台酒款管理。
7. 后台码池管理。
8. 基础扫码数据统计。

### 13.2 建议后置的内容

1. 复杂会员体系。
2. 抽奖、红包、积分商城。
3. 防窜货物流链路。
4. AR、3D、视频大片式内容。
5. 多语言与全球站点体系。

## 14. 里程碑建议

### 14.1 第一阶段：验证版本

目标：在真实物料上跑通“扫码进入内容体验”的闭环。

交付建议：

1. 1 至 3 款酒的内容模板。
2. 一批测试二维码。
3. 小程序和 H5 双入口。
4. 基础后台和扫码日志。

### 14.2 第二阶段：品牌化版本

目标：提升视觉表现、内容密度和品牌资产沉淀能力。

交付建议：

1. 完整品牌主页。
2. 更强的酒款详情模板。
3. 分享卡片与活动皮肤。
4. 音乐内容的正式授权方案。

### 14.3 第三阶段：运营化版本

目标：让扫码入口具备持续复用的商业价值。

交付建议：

1. 会员注册或留资。
2. 优惠券、活动、礼赠关系。
3. 真伪验证与客服联动。
4. 数据看板和渠道分析。

## 15. 需要甲方进一步确认的问题

为了进入 PRD 和设计阶段，建议尽快确认以下事项：

1. 目标用户是否几乎全部位于微信生态内。
2. 二维码绑定对象是“每瓶酒”“每盒礼盒”还是“每个订单”。
3. 一次性规则是否接受“首次激活绑定”而不是“绝对失效”。
4. 音乐内容是自有版权、采购版权，还是跳转第三方平台。
5. 是否需要把扫码入口兼作真伪查询入口。
6. 是否计划在第二阶段承接会员、优惠券或复购。
7. 是否需要支持海外访客或多语言内容。

## 16. 最终建议

从当前需求看，本项目最合适的落地方式不是“纯内容网站”，也不是“重防伪系统”，而是一个`以微信小程序为主、兼容网页访问的一物一码品牌体验产品`。

建议的总体判断如下：

1. 入口层：小程序优先，H5 兜底。
2. 二维码层：统一短链/令牌管理，不把业务规则直接写死在二维码里。
3. 一次性策略：采用首次激活绑定型规则。
4. 内容层：优先打磨酒款介绍与音乐体验，而不是一开始就堆复杂营销玩法。
5. 系统层：后台必须具备内容配置、码池管理、状态识别和数据统计能力。

如果目标是先快速验证市场与客户接受度，这一方案的投入、风险和可扩展性相对平衡，适合作为 MVP 启动。
