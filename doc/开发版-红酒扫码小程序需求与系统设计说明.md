# 红酒扫码小程序需求与系统设计说明（开发版）

- 文档版本：v2.0
- 编写日期：2026-04-15
- 适用对象：产品、研发、测试、运维、项目经理
- 文档目标：基于当前代码底座，明确完整项目需求、阶段边界、系统设计与实施风险
- 版本变更：v1.0 → v2.0，核心变更——“一物一码”模式改为”固定二维码 + 六位提取码”模式

## 1. 文档结论先行

当前仓库已经实现了`扫码进入 + 小程序内容体验 + 音乐播放`的 MVP 底座，但距离用户提出的完整业务目标，还缺少以下三个系统级模块：

1. `商城交易系统`
商品、价格、库存、购物车、订单、支付、履约。

2. `后台运营系统`
管理员登录、内容配置、提取码管理、商品管理、价格管理、数据看板。

3. `会员与数字权益系统`
会员购买、权益发放、音乐无限听、付费下载链接、下载审计、风控与合规。

### v2.0 核心变更说明

本次需求变更为：将原先的”每瓶酒生成唯一二维码”模式，调整为**”固定小程序码 + 六位提取码”**模式。

变更要点：

1. **入口方式**：不再为每瓶酒生成独立二维码。改为提供一张固定的小程序码，所有用户扫同一个码进入。
2. **验证方式**：用户扫码进入后，需手动输入六位数字提取码。提取码由管理员在后台批量生成并配置。
3. **一次性消费**：每个提取码只能使用一次。一旦被某位用户输入并验证成功，该码立即失效，不可重复使用。
4. **内容绑定**：一个提取码绑定一瓶酒的介绍内容 + 一首歌曲。用户输入正确提取码后，即可查看该酒的介绍、酒庄信息和专属歌曲。
5. **音乐循环播放**：歌曲在用户未退出小程序期间循环播放。用户退出小程序后，播放结束，再次进入需要输入新的提取码。
6. **后台配置**：管理员可在后台创建提取码批次，并配置每个提取码对应的酒款和歌曲。

## 2. 当前代码基线梳理

### 2.1 已实现模块

当前仓库中，已经存在以下能力：

1. `微信小程序前端`
页面包括扫码进入页、酒窖页、酒款详情页、音乐页。

2. `服务端接口`
已提供健康检查、创建演示码、创建一次性码、消费一次性码、按 session 恢复体验数据等接口。

3. `提取码状态管理`
当前通过 JSON 存储 `wines`、`codes`、`sessions` 三类数据，已支持首次使用成功、重复使用拒绝、过期拒绝。（注：v1.0 中为"一次性二维码"，v2.0 调整为"六位提取码"，数据结构类似但入口逻辑不同。）

4. `音乐播放`
小程序内使用音频上下文播放本地示例音频资源。

5. `真实微信小程序码接入预留`
服务端已封装生成微信小程序码的逻辑，可在配置 AppID 和 AppSecret 后接入真实环境。

### 2.2 当前技术边界

当前代码更偏向`演示原型`而非正式商用系统，具体边界如下：

1. 数据库存储尚未引入，仍使用 JSON 文件。
2. 没有用户体系。
3. 没有管理员鉴权。
4. 没有商城、订单、支付。
5. 没有会员体系。
6. 没有数字下载能力。
7. 没有运维日志、告警、监控与审计。
8. 入口仍为"一物一码"模式，尚未改为"固定码 + 提取码"模式。

### 2.3 对当前代码的建议定位

建议将当前代码视为：

`P0：品牌体验验证底座`

而不要把它当作可直接商用上线的完整系统。

## 3. 完整业务目标

用户提出的目标可以拆成 4 个业务域：

1. `提取码专属体验域`
用户扫描固定小程序码进入，输入六位数字提取码，即可查看对应的酒庄介绍、酒款介绍和专属音乐体验。一个提取码对应一瓶酒 + 一首歌曲。提取码为一次性消费，使用后立即失效。

2. `商城交易域`
用户从小程序常规入口进入后，可以浏览商城、查看酒庄介绍、购买多种酒品。

3. `后台运营域`
管理员可以配置提取码对应的酒款、酒庄介绍、歌曲内容，以及商城商品、价格与上下架状态。

4. `会员权益域`
用户支付后可获得下载歌曲的权益，并成为会员；会员再次进入小程序时可以无限听歌并拥有更完整的访问权限。

## 7. 核心业务流程设计

### 7.1 流程 A：用户输入提取码进入专属体验

1. 用户扫描固定的小程序码（可印在酒瓶、礼盒或宣传物料上）。
2. 小程序启动，进入提取码输入页。
3. 用户输入六位数字提取码，点击验证。
4. 前端调用 `POST /api/redeem/consume` 接口，传入提取码。
5. 服务端校验提取码是否存在、是否过期、是否已使用。
6. 若验证成功：
   - 将提取码标记为已使用（`claimed`），绑定当前用户。
   - 返回该码对应的酒款、酒庄、歌曲等体验数据。
7. 前端进入专属体验区，展示酒款介绍、酒庄信息和歌曲播放器。
8. 歌曲自动播放并循环，直至用户退出小程序。
9. 若输入已使用、不存在或已过期的提取码：
   - 返回”提取码无效”或”提取码已使用”的提示。
10. 用户退出小程序后，体验结束。再次进入需输入新的提取码。

### 7.2 流程 B：用户正常进入小程序商城

1. 用户从小程序首页、分享卡片或收藏入口进入。
2. 前端加载品牌主页、酒庄介绍与商城推荐。
3. 用户浏览商品、加入购物车、提交订单。
4. 支付完成后生成订单与履约记录。

### 7.3 流程 C：管理员配置提取码与内容

1. 管理员登录后台。
2. 新增或编辑酒款。
3. 配置酒庄介绍、酒款详情、歌曲资源、封面图。
4. 创建提取码批次：
   - 指定批次名称、数量。
   - 选择该批次对应的酒款和歌曲。
   - 系统批量生成六位数字提取码。
5. 导出提取码列表，分配给印刷或包装环节。
6. 查看每个提取码的使用状态、使用时间和使用用户。

### 7.4 流程 D：用户购买会员与下载歌曲

### 7.4 流程 D：用户购买会员与下载歌曲

1. 用户在歌曲页点击“开通会员 / 购买下载”。
2. 系统展示权益说明。
3. 用户支付成功。
4. 系统发放会员权益与下载权益。
5. 用户可无限播放歌曲。
6. 用户点击下载时，系统生成签名下载链接。
7. 系统记录下载行为、设备信息、时间和次数。

## 9. 状态机设计

1. `ready`

## 12. 技术架构建议

### 12.1 推荐架构

建议使用以下结构：

1. `微信小程序前端`
消费者使用。

2. `后台 Web 管理端`
运营、商品、客服、管理员使用。

3. `API 服务`
统一承载业务接口、权限校验和支付回调。

4. `关系型数据库`
推荐 MySQL 或 PostgreSQL。

5. `Redis`
缓存 session、验证码、签名令牌、限流计数。

6. `对象存储 + CDN`
存图片、音频、下载文件、二维码图。

7. `异步任务队列`
处理提取码批量生成、下载链接签发、统计汇总、消息通知。

### 12.2 为什么当前 JSON 存储必须升级

当前 JSON 文件存储只适合演示，正式版不适合继续沿用，原因包括：

1. 并发写入不安全。
2. 无法支持复杂查询。
3. 无法支撑后台管理筛选和分页。
4. 无法支撑订单和支付回调一致性。
5. 无法支撑操作审计。

## 13. 权限与可见性设计

### 13.1 提取码访客权限

1. 可看当前提取码对应的酒款内容。
2. 可看酒庄介绍。
3. 可播放该码绑定的歌曲（循环播放直至退出小程序）。
4. 不可看商城。
5. 不可浏览其他酒款。
6. 退出小程序后体验结束，需重新输入提取码。

### 13.2 普通用户权限

1. 可看公共酒庄介绍。
2. 可看商城和商品详情。
3. 可下单购买实物酒。
4. 歌曲播放受试听规则限制。

### 13.3 会员权限

1. 具备普通用户所有权限。
2. 可无限播放会员可用歌曲。
3. 可查看会员中心。
4. 可获得下载权益。

## 14. 支付与数字下载设计重点

### 14.1 建议不要把“付费下载”直接理解为简单文件下载

该需求实质上包含三部分：

1. 支付。
2. 权益发放。
3. 文件交付。

任何一部分没设计好，都会出现投诉或审核风险。

### 14.2 推荐的实现方式

建议将“买歌下载”设计成：

1. 用户购买`会员包`或`数字权益包`。
2. 支付成功后，系统发放会员权益和下载权益。
3. 用户在会员中心或歌曲页点击下载。
4. 服务端生成短时有效的签名链接。
5. 下载行为计数，并支持后台查询。

### 14.3 关于“小程序下载到本地”的实现边界

需要特别说明：

1. 小程序内可以完成文件下载与本地缓存。
2. 但“真正保存到用户手机公共文件系统并可被外部播放器长期使用”的体验，在不同系统与审核环境下边界不完全一致。
3. 因此正式立项时，建议把验收口径定义为：
`用户支付后可获得可控下载权益，并能在小程序内完成下载或离线保存。`
4. 如果客户要求“必须导出到手机文件管理器 / 其他播放器可直接识别”，建议补充 H5 下载页或 App 方案作为兜底。

### 14.4 虚拟权益支付合规

截至 2026-04-11 可查资料显示，微信生态对虚拟支付能力的支持正在增强，但不同类目、终端和审核口径仍可能存在差异。

因此开发原则应是：

1. 不在立项阶段默认认为“数字音乐下载”一定可以像普通实物一样直接售卖。
2. 项目启动前必须用当前主体资质、类目和微信支付后台权限做一次正式核验。
3. 方案上应保留“会员权益包优先、单曲下载为可选能力”的回退空间。

## 15. 后台管理设计重点

### 15.1 酒款内容管理

管理员应能维护：

1. 酒款名称、年份、产区、酒精度。
2. 酒庄介绍、图文说明。
3. 风味描述、醒酒建议、配餐建议。
4. 封面图、展示图、海报图。
5. 绑定歌曲或音频资源。

### 15.2 提取码管理

管理员应能：

1. 创建提取码批次：指定酒款、歌曲、生成数量。
2. 系统自动批量生成六位数字提取码。
3. 导出提取码列表（CSV 或 Excel）。
4. 查看每个提取码的使用状态、使用时间和使用用户。
5. 作废异常提取码。
6. 查看高频失败尝试或异常使用行为。

### 15.3 商品与价格管理

管理员应能：

1. 配置商品。
2. 配置 SKU。
3. 修改价格。
4. 设置库存。
5. 上下架商品。

### 15.4 会员与客服

管理员应能：

1. 查看用户是否已成为会员。
2. 查看下载权益是否已发放。
3. 人工补发会员权益。
4. 处理下载失败、提取码异常、会员争议。

## 16. 埋点与数据分析设计

建议在正式版加入以下埋点：

1. 提取码输入次数（含成功与失败）。
2. 提取码验证成功率。
3. 提取码失效率。
4. 酒款详情浏览量。
5. 音乐播放次数与完成率。
6. 商城点击率。
7. 下单转化率。
8. 会员购买转化率。
9. 下载次数与失败率。

## 17. 安全与风控设计

### 17.1 提取码风控

1. 提取码生成应随机且不连续，避免被猜测（如不使用纯递增数字）。
2. 同一提取码高频验证失败需限流（如同一 IP/设备连续输错 5 次后暂停 5 分钟）。
3. 提取码验证接口需防暴力破解（六位数字仅有 100 万种组合，建议加入频率限制和验证码机制）。
4. 支持异常设备或异常地区告警。

### 17.2 下载风控

1. 下载链接必须短时有效。
2. 同一权益的下载次数应可限制。
3. 下载日志必须可追踪。

### 17.3 后台安全

1. 后台必须做 RBAC 权限。
2. 关键操作写审计日志。
3. 支持账号禁用和密码重置。

## 18. 合规与风险清单

### 18.1 酒类展示合规

需注意：

1. 未成年人不可引导饮酒。
2. 页面需提供理性饮酒提示。
3. 不可出现过度诱导饮酒文案。

### 18.2 音乐版权

如果歌曲不是自有版权或明确授权，不建议直接提供在线播放与下载。

### 18.3 隐私与支付

若接入登录、手机号、支付、订单、会员，需要补齐：

1. 用户协议。
2. 隐私政策。
3. 支付后的售后说明。
4. 下载权益说明。

## 19. 推荐实施顺序

### 阶段一：内容正式化

1. 将入口改为"固定小程序码 + 提取码输入"模式。
2. 真实酒款与酒庄内容接入。
3. 提取码批次管理（后台生成、导出、分配）。
4. 后台内容配置（酒款、酒庄、歌曲绑定）。
5. 提取码一次性消费与失效机制。
6. 音乐循环播放机制。

### 阶段二：商城上线

1. 商品、价格、库存。
2. 订单与支付。
3. 管理后台商品管理。

### 阶段三：会员与数字权益

1. 会员购买。
2. 无限听歌。
3. 下载权益。
4. 下载日志与客服处理。

## 20. 验收建议

### 20.1 P1 验收

1. 用户扫描固定小程序码后进入提取码输入页。
2. 输入有效提取码后能查看对应的酒款内容、酒庄信息和歌曲。
3. 歌曲循环播放直至用户退出小程序。
4. 同一提取码重复使用被正确拦截。
5. 输入无效提取码给出明确提示。
6. 后台可批量生成提取码并导出。
7. 后台可配置提取码对应的酒款与歌曲。
8. 后台可查看提取码使用状态。

### 20.2 P2 验收

1. 普通用户可浏览商城并下单。
2. 管理员可修改商品和价格。
3. 订单、支付、后台状态一致。

### 20.3 P3 验收

1. 支付后用户自动成为会员。
2. 会员可无限听歌。
3. 用户可获得下载权益。
4. 后台可查询下载记录与权益状态。

## 21. 提取码设计补充说明

### 21.1 为什么选择六位提取码

1. 六位数字（000000–999999）提供 100 万种组合，对当前业务规模足够。
2. 六位数字便于用户手动输入，不会因过长导致体验差。
3. 配合一次性消费和频率限制机制，安全性可满足需求。

### 21.2 提取码生成规则建议

1. 系统生成时应保证随机性，避免连续递增（如不要生成 123456、123457、123458...）。
2. 建议使用不可预测的随机算法生成。
3. 同一批次内提取码不可重复。
4. 全局提取码不可重复（跨批次也要保证唯一）。
5. 可考虑为不同批次设置不同前缀以方便区分（如 A 批次为 1XXXXX，B 批次为 2XXXXX），但前缀位数不超过 1 位。

### 21.3 提取码分配与物理载体

1. 提取码生成后导出为列表。
2. 每个提取码印制在酒瓶标签、礼盒卡片或宣传物料上。
3. 用户购买/收到酒后，刮开涂层或翻开卡片即可看到提取码。
4. 扫描固定小程序码 → 输入提取码 → 体验专属内容。
