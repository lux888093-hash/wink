# 红酒扫码小程序需求与系统设计说明（开发版）

- 文档版本：v1.0
- 编写日期：2026-04-11
- 适用对象：产品、研发、测试、运维、项目经理
- 文档目标：基于当前代码底座，明确完整项目需求、阶段边界、系统设计与实施风险

## 1. 文档结论先行

当前仓库已经实现了`扫码一次性核销 + 小程序内容体验 + 音乐播放 + 会话恢复`的 MVP 底座，但距离用户提出的完整业务目标，还缺少以下三个系统级模块：

1. `商城交易系统`
商品、价格、库存、购物车、订单、支付、履约。

2. `后台运营系统`
管理员登录、内容配置、码池管理、商品管理、价格管理、数据看板。

3. `会员与数字权益系统`
会员购买、权益发放、音乐无限听、付费下载链接、下载审计、风控与合规。

因此，本项目的正确开发策略不是在当前代码上“零散补功能”，而是围绕现有扫码体验底座，补齐完整的数据模型、权限体系和服务边界。

## 2. 当前代码基线梳理

### 2.1 已实现模块

当前仓库中，已经存在以下能力：

1. `微信小程序前端`
页面包括扫码进入页、酒窖页、酒款详情页、音乐页。

2. `服务端接口`
已提供健康检查、创建演示码、创建一次性码、消费一次性码、按 session 恢复体验数据等接口。

3. `一次性码状态管理`
当前通过 JSON 存储 `wines`、`codes`、`sessions` 三类数据，已支持首次扫码成功、重复扫码拒绝、过期拒绝。

4. `音乐播放`
小程序内使用音频上下文播放本地示例音频资源。

5. `真实微信小程序码接入预留`
服务端已封装生成微信小程序码的逻辑，可在配置 AppID 和 AppSecret 后接入真实环境。

### 2.2 当前技术边界

当前代码更偏向`演示原型`而非正式商用系统，具体边界如下：

1. 数据库存储尚未引入，仍使用 JSON 文件。
2. 没有用户体系。
3. 没有管理员鉴权。
4. 没有商城、订单、支付。
5. 没有会员体系。
6. 没有数字下载能力。
7. 没有运维日志、告警、监控与审计。

### 2.3 对当前代码的建议定位

建议将当前代码视为：

`P0：品牌体验验证底座`

而不要把它当作可直接商用上线的完整系统。

## 3. 完整业务目标

用户提出的目标可以拆成 4 个业务域：

1. `扫码专属体验域`
围绕某一瓶酒生成唯一二维码，扫码后进入酒庄介绍、酒款介绍和专属音乐体验。

2. `商城交易域`
用户从小程序常规入口进入后，可以浏览商城、查看酒庄介绍、购买多种酒品。

3. `后台运营域`
管理员可以配置扫码对应的酒、酒庄介绍、歌曲内容、商城商品、价格与上下架状态。

4. `会员权益域`
用户支付后可获得下载歌曲的权益，并成为会员；会员再次进入小程序时可以无限听歌并拥有更完整的访问权限。

## 4. 角色定义

### 4.1 C 端角色

1. `扫码访客`
通过某瓶酒上的二维码首次进入，只能看该酒相关内容，不显示商城。

2. `普通用户`
直接进入小程序，可看酒庄介绍和商城，可浏览商品并下单。

3. `会员用户`
拥有普通用户能力，并额外具备无限听歌、下载权益、会员专区等能力。

### 4.2 B 端角色

1. `运营管理员`
维护酒庄文案、酒款信息、图片、歌曲绑定、二维码批次。

2. `商品管理员`
维护商品、价格、库存、上下架。

3. `客服管理员`
处理异常码、会员权益争议、下载失败和订单售后。

4. `超级管理员`
拥有角色授权、系统配置、支付配置、存储配置和风控开关权限。

## 6. 产品信息架构

### 6.1 小程序前台模块

建议前台划分为以下模块：

1. `扫码落地页`
接收二维码 scene 或 token，校验是否首次扫码，创建专属体验会话。

2. `专属酒款体验区`
仅扫码访客可见。展示该瓶酒的酒庄内容、酒款介绍、专属音乐，不展示商城入口。

3. `公共首页`
普通用户和会员用户可见。展示酒庄品牌介绍、精选商品、会员入口。

4. `商城模块`
商品列表、商品详情、购物车、下单、支付。

5. `音乐模块`
播放歌曲、判断普通用户试听限制、会员无限播放状态、购买下载入口。

6. `会员中心`
会员状态、权益说明、订单记录、下载记录、客服入口。

### 6.2 后台模块

建议后台划分为以下模块：

1. `登录与权限`
管理员登录、角色权限、操作日志。

2. `酒庄与酒款内容管理`
酒庄、酒款、图文、音频、绑定关系管理。

3. `二维码管理`
批次生成、导出、状态查询、作废、异常处理。

4. `商品与价格管理`
SKU、价格、库存、上下架、活动价。

5. `订单管理`
订单列表、支付状态、发货状态、退款状态。

6. `会员与权益管理`
会员开通记录、权益发放、下载权限、人工补发。

7. `统计分析`
扫码次数、首扫率、会员转化、下单转化、下载次数。

## 7. 核心业务流程设计

### 7.1 流程 A：用户扫码进入专属体验

1. 用户扫描酒瓶或礼盒上的二维码。
2. 小程序启动并带入 `scene/token`。
3. 前端调用 `redeem/consume` 接口。
4. 服务端校验二维码是否存在、是否过期、是否已用。
5. 若首次扫码成功：
生成 `scan_session`，绑定到该酒款和该次会话。
6. 返回酒款、酒庄、歌曲等体验数据。
7. 前端进入专属体验区，不展示商城。
8. 后续重复扫码同一码时：
返回“该码已使用”或“该码已绑定”的提示。

### 7.2 流程 B：用户正常进入小程序商城

1. 用户从小程序首页、分享卡片或收藏入口进入。
2. 前端加载品牌主页、酒庄介绍与商城推荐。
3. 用户浏览商品、加入购物车、提交订单。
4. 支付完成后生成订单与履约记录。

### 7.3 流程 C：管理员配置扫码内容

1. 管理员登录后台。
2. 新增或编辑酒款。
3. 配置酒庄介绍、酒款详情、歌曲资源、封面图。
4. 创建二维码批次，并绑定到指定酒款或活动批次。
5. 导出二维码给印刷或包装环节。

### 7.4 流程 D：用户购买会员与下载歌曲

1. 用户在歌曲页点击“开通会员 / 购买下载”。
2. 系统展示权益说明。
3. 用户支付成功。
4. 系统发放会员权益与下载权益。
5. 用户可无限播放歌曲。
6. 用户点击下载时，系统生成签名下载链接。
7. 系统记录下载行为、设备信息、时间和次数。

## 9. 状态机设计

### 9.1 二维码状态

建议 `scan_codes.status` 使用以下枚举：

1. `ready`
未使用，可首扫。

2. `claimed`
已首扫并绑定。

3. `expired`
已过期。

4. `disabled`
人工停用。

相比当前代码里的 `ready/used` 两态，正式版需要扩展为可管理状态机。

### 9.3 会员状态

## 12. 技术架构建议

### 12.1 推荐架构

建议使用以下结构：

1. `微信小程序前端`
消费者使用。

2. `后台 Web 管理端`
运营、商品、客服、管理员使用。

3. `API 服务`
统一承载业务接口、权限校验和支付回调。

4. `关系型数据库`
推荐 MySQL 或 PostgreSQL。

5. `Redis`
缓存 session、验证码、签名令牌、限流计数。

6. `对象存储 + CDN`
存图片、音频、下载文件、二维码图。

7. `异步任务队列`
处理二维码批量生成、下载链接签发、统计汇总、消息通知。

### 12.2 为什么当前 JSON 存储必须升级

当前 JSON 文件存储只适合演示，正式版不适合继续沿用，原因包括：

1. 并发写入不安全。
2. 无法支持复杂查询。
3. 无法支撑后台管理筛选和分页。
4. 无法支撑订单和支付回调一致性。
5. 无法支撑操作审计。

## 14. 支付与数字下载设计重点

### 14.1 建议不要把“付费下载”直接理解为简单文件下载

该需求实质上包含三部分：

1. 支付。
2. 权益发放。
3. 文件交付。

任何一部分没设计好，都会出现投诉或审核风险。

### 14.2 推荐的实现方式

建议将“买歌下载”设计成：

1. 用户购买`会员包`或`数字权益包`。
2. 支付成功后，系统发放会员权益和下载权益。
3. 用户在会员中心或歌曲页点击下载。
4. 服务端生成短时有效的签名链接。
5. 下载行为计数，并支持后台查询。

### 14.3 关于“小程序下载到本地”的实现边界

需要特别说明：

1. 小程序内可以完成文件下载与本地缓存。
2. 但“真正保存到用户手机公共文件系统并可被外部播放器长期使用”的体验，在不同系统与审核环境下边界不完全一致。
3. 因此正式立项时，建议把验收口径定义为：
`用户支付后可获得可控下载权益，并能在小程序内完成下载或离线保存。`
4. 如果客户要求“必须导出到手机文件管理器 / 其他播放器可直接识别”，建议补充 H5 下载页或 App 方案作为兜底。

### 14.4 虚拟权益支付合规

截至 2026-04-11 可查资料显示，微信生态对虚拟支付能力的支持正在增强，但不同类目、终端和审核口径仍可能存在差异。

因此开发原则应是：

1. 不在立项阶段默认认为“数字音乐下载”一定可以像普通实物一样直接售卖。
2. 项目启动前必须用当前主体资质、类目和微信支付后台权限做一次正式核验。
3. 方案上应保留“会员权益包优先、单曲下载为可选能力”的回退空间。

## 15. 后台管理设计重点

### 15.1 酒款内容管理

管理员应能维护：

1. 酒款名称、年份、产区、酒精度。
2. 酒庄介绍、图文说明。
3. 风味描述、醒酒建议、配餐建议。
4. 封面图、展示图、海报图。
5. 绑定歌曲或音频资源。

### 15.2 二维码管理

管理员应能：

1. 按酒款或批次生成二维码。
2. 导出二维码图片包。
3. 查看每个码的首次扫码时间、状态和用户。
4. 作废异常码。
5. 查看重复扫码或异常频次。

### 15.3 商品与价格管理

管理员应能：

1. 配置商品。
2. 配置 SKU。
3. 修改价格。
4. 设置库存。
5. 上下架商品。

### 15.4 会员与客服

管理员应能：

1. 查看用户是否已成为会员。
2. 查看下载权益是否已发放。
3. 人工补发会员权益。
4. 处理下载失败、码异常、会员争议。

## 16. 埋点与数据分析设计

建议在正式版加入以下埋点：

1. 扫码次数。
2. 首扫成功率。
3. 码失效率。
4. 酒款详情浏览量。
5. 音乐播放次数与完成率。
6. 商城点击率。
7. 下单转化率。
8. 会员购买转化率。
9. 下载次数与失败率。

## 17. 安全与风控设计

### 17.1 扫码风控

1. token 不可预测。
2. 同码高频请求需限流。
3. 支持异常设备或异常地区告警。

### 17.2 下载风控

1. 下载链接必须短时有效。
2. 同一权益的下载次数应可限制。
3. 下载日志必须可追踪。

### 17.3 后台安全

1. 后台必须做 RBAC 权限。
2. 关键操作写审计日志。
3. 支持账号禁用和密码重置。

## 18. 合规与风险清单

### 18.1 酒类展示合规

需注意：

1. 未成年人不可引导饮酒。
2. 页面需提供理性饮酒提示。
3. 不可出现过度诱导饮酒文案。

### 18.2 音乐版权

如果歌曲不是自有版权或明确授权，不建议直接提供在线播放与下载。

### 18.3 隐私与支付

若接入登录、手机号、支付、订单、会员，需要补齐：

1. 用户协议。
2. 隐私政策。
3. 支付后的售后说明。
4. 下载权益说明。

## 19. 推荐实施顺序

### 阶段一：内容正式化

1. 真实酒款与酒庄内容接入。
2. 二维码批次管理。
3. 后台内容配置。

### 阶段二：商城上线

1. 商品、价格、库存。
2. 订单与支付。
3. 管理后台商品管理。

### 阶段三：会员与数字权益

1. 会员购买。
2. 无限听歌。
3. 下载权益。
4. 下载日志与客服处理。

## 20. 验收建议

### 20.1 P1 验收

1. 扫码后能进入指定酒款内容。
2. 同一码重复扫码被正确拦截。
3. 后台可修改酒款与歌曲内容。

### 20.2 P2 验收

1. 普通用户可浏览商城并下单。
2. 管理员可修改商品和价格。
3. 订单、支付、后台状态一致。

### 20.3 P3 验收

1. 支付后用户自动成为会员。
2. 会员可无限听歌。
3. 用户可获得下载权益。
4. 后台可查询下载记录与权益状态。
