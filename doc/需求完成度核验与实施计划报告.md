# 红酒扫码小程序需求完成度核验与实施计划报告

- 核验日期：2026-04-16
- 核验对象：
  - `doc/开发版-红酒扫码小程序需求与系统设计说明.md`
  - `doc/项目现状分析与实施计划.md`
  - 当前仓库代码：`miniprogram/`、`server/`、后台静态页
- 核验方式：需求逐项对照、代码静态审查、关键函数验证、官方资料交叉参考
- 结论等级说明：
  - `完成`：功能闭环满足需求，且具备基本验收条件。
  - `部分完成`：有页面/API/数据结构，但业务语义、生产能力或关键边界不满足。
  - `未完成`：代码中没有实现，或当前实现无法按需求运行。
  - `风险`：功能看似存在，但会导致错误、安全问题、合规问题或上线阻塞。

## 1. 总体结论

当前项目不是“95% 完成”的可上线版本。更准确的定位是：

> 已经具备较完整的本地演示原型和页面/API骨架，但距离需求文档定义的正式商用系统仍有明显差距。

保守评估：

| 维度 | 当前完成度判断 | 说明 |
|---|---:|---|
| 演示原型完成度 | 约 65%-70% | 页面、API、演示数据、mock 支付、会员/下载演示链路基本存在。 |
| 正式业务验收完成度 | 约 45%-55% | 多个核心需求语义不满足，包括提取码绑定歌曲、批量生成、退出失效、真实支付、真实数据库、RBAC。 |
| 商用上线完成度 | 约 35%-45% | 生产配置、交易合规、对象存储、监控告警、测试、权限、数据一致性仍缺。 |

现状文档中“阶段二、三、四完成”“当前项目完成度约 95%”的判断偏乐观，不能作为上线依据。

## 3. 核心阻塞问题

### 3.1 提取码批量生成功能实际不可用

现状文档把“提取码批次管理”标为已完成，但代码里 `createCodeBatch` 存在运行时错误：

- 文件：`server/services/store.js`
- 位置：`createCodeBatch` 内部生成 `codes` 的回调
- 问题：在 `const codes = ...` 初始化过程中引用了 `codes` 自身。
- 验证结果：直接调用 `createCodeBatch({ quantity: 2 })` 报错：

```text
ReferenceError: Cannot access 'codes' before initialization
```

影响：

- 后台“生成提取码批次”无法验收。
- P1 的“批次生成、导出、分配”核心功能不能算完成。
- 如果不修复，正式运营无法批量生产酒瓶/礼盒提取码。

建议修复：

- 使用局部数组逐个 push，或先建 `generatedCodes = []`。
- 每次生成时使用 `store.scanCodes + generatedCodes` 做去重。
- 增加单元测试覆盖 `quantity=1/2/500`、批次号重复、全局重复码、过期时间。

### 3.2 提取码没有绑定“一首歌曲”

需求 v2.0 明确写明：

- 一个提取码绑定一瓶酒的介绍内容 + 一首歌曲。
- 管理员创建提取码批次时配置该批次对应的酒款和歌曲。

当前实现：

- `scanCodes` 只有 `wineId`，没有 `trackId`。
- `createCodeBatch` 后台表单只有“绑定酒款”，没有“绑定歌曲”。
- `consumeOneTimeCode` 生成 session 时使用该酒款的全部 `trackIds`。
- `buildWineExperience` 返回该酒款下的全部曲目。

影响：

- “一个提取码对应一瓶酒 + 一首歌曲”没有实现。
- 扫码访客可能获得该酒款下多首曲目的完整播放权限。
- 后台无法按批次绑定不同歌曲，运营需求不满足。

建议修复：

- `scanCodes` 增加 `trackId`。
- `codeBatches` 增加 `trackId`。
- `POST /api/admin/code-batches` 必须校验 `wineId + trackId`，且 `track.wineId === wineId`。
- `consumeOneTimeCode` 的 `scopeJson.trackIds` 只放当前 `trackId`。
- 后台“生成提取码批次”增加曲目下拉选择。
- CSV 导出增加歌曲字段。

### 3.3 “退出小程序后体验结束”未严格实现

需求要求：

- 用户退出小程序后播放结束。
- 再次进入需要输入新的提取码。
- 不需要会话恢复。

当前实现：

- 服务端仍保留 `scanSessions`。
- `GET /api/sessions/:sessionId` 仍可恢复体验。
- 首页 `getStoreHome` 会返回未过期 `exclusiveEntry`。
- 小程序首页显示“首扫记录已保留”，可继续进入专属体验。

影响：

- 当前行为与 v2.0 需求冲突。
- 如果业务坚持“退出即结束”，需要移除或关闭恢复入口。
- 如果产品希望保留 30 分钟容错，则需求文档需要修改，明确“短时恢复”规则。

建议二选一：

1. 严格按需求：消费成功后只在内存保留体验，不持久化恢复入口，后台仅记录使用状态。
2. 产品改口：允许 30 分钟内同一用户恢复，并把验收标准改成“退出后音乐停止，体验页 30 分钟内可继续进入”。

### 3.4 音乐“自动播放”没有实现，播放 UI 也未绑定真实曲目信息

需求要求：

- 验证成功后进入专属体验。
- 歌曲自动播放并循环，直到用户退出。

当前实现：

- `setupAudio(currentTrack)` 默认不自动播放，只有切歌时传 `autoplay=true`。
- `audioContext.loop = true` 已设置，所以“播放后循环”存在。
- `melody/index.wxml` 中标题、封面、描述仍是硬编码的 `Vintage Noir`、固定封面和固定描述，没有展示 `currentTrack.cnTitle`、`currentTrack.cover`、`currentTrack.description`。
- 进度条标签也是固定值，未真实显示当前播放时间和总时长。

影响：

- “自动播放并循环”只能算部分完成。
- 用户看到的歌曲信息可能和实际播放音频不一致。
- 会员/试听/解锁相关提示在音乐页 JS 中有方法，但 WXML 没有入口，体验不闭环。

建议修复：

- 进入专属音乐页后尝试自动播放，并兼容微信对自动播放的限制：自动播放失败时显示明确的一键播放按钮。
- WXML 全部改为绑定 `currentTrack`。
- 真实更新 `currentTimeLabel`、`durationLabel`、`progress`。
- 在音乐页显示试听限制、会员状态、解锁/下载入口，或明确把这些功能只放在会员中心。

### 3.5 固定小程序码入口没有形成交付物

当前代码有 `server/services/wechat.js`，封装了 `getwxacodeunlimit` 逻辑，`server/index.js` 也设置了 `pagePath = 'pages/redeem/index'`，但没有看到实际调用入口。

当前缺口：

- 没有后台按钮或 API 生成固定小程序码。
- 没有固定码图片交付物。
- 没有生产 AppID/AppSecret 配置验证。
- 没有说明该固定码是否由微信公众平台后台生成，还是由服务端生成。

建议：

- 如果固定码永远只指向 `pages/redeem/index`，优先用微信公众平台生成并留档。
- 如果需要服务端生成，增加 `POST /api/admin/qrcode/fixed-redeem`，生成后在后台展示下载链接。
- 小程序码生成结果应纳入上线验收，不能只保留未调用的服务函数。

### 3.6 会员购买和单曲解锁绕过真实支付

当前实现：

- `purchaseMembership` 直接创建已支付虚拟订单。
- `unlockTrack` 直接创建已支付数字权益订单。
- 前端点击“立即开通”或“解锁”后，没有走 `POST /api/payments/orders/:orderId/jsapi`。

影响：

- P3 “用户支付后成为会员”“付费下载歌曲”不能按正式交易验收。
- 当前只能算演示流程。
- 订单、支付、权益发放的一致性没有在真实微信支付链路中验证。

建议：

- 会员购买、单曲解锁统一改为“创建 pending_payment 订单 -> JSAPI 支付 -> 回调/查单确认 -> 发放权益”。
- 权益发放必须只由支付成功后的服务端逻辑触发，不能由前端按钮直接触发。
- 增加幂等键，避免重复支付或重复发放权益。

### 3.7 数据存储不是正式关系型数据库模型

README 中写“PostgreSQL 兼容存储层”，但当前 `db-postgres-worker.js` 实际只读写 `app_state.store_snapshot` 一个 JSONB 快照。

影响：

- schema 中虽然有 `wines`、`orders`、`payments` 等表，但运行逻辑未按这些表做增删改查。
- 无法依赖数据库唯一索引保证提取码唯一。
- 无法用事务保证“一次性消费”“库存扣减”“支付回调权益发放”的一致性。
- 并发写入存在覆盖风险。
- 后台筛选、分页、审计、报表能力会受限。

建议：

- 进入正式版前，必须把核心实体迁入真实表：
  - `redeem_codes`
  - `redeem_code_batches`
  - `users`
  - `wines`
  - `tracks`
  - `products`
  - `product_skus`
  - `orders`
  - `payments`
  - `memberships`
  - `download_entitlements`
  - `download_logs`
  - `admin_users`
  - `audit_logs`
- 提取码消费使用事务和行级锁，或唯一状态更新：

```sql
update redeem_codes
set status = 'claimed', used_at = now(), used_by_user_id = $user_id
where code = $code and status = 'ready' and expires_at > now()
returning *;
```

### 3.8 后台 RBAC 只存在数据，没有权限执行

当前种子数据有 `adminRoles`，管理员登录也返回 `roleId`，但：

- `withAdmin` 只判断是否登录。
- 所有后台接口登录后都可访问。
- 没有按权限点限制读写。
- 后台没有角色管理、管理员管理、禁用/重置流程。

影响：

- B 端角色定义中的“运营管理员、商品管理员、客服管理员、超级管理员”没有真正实现。
- 审计里很多操作 actor 固定为默认管理员，不利于追责。

建议：

- 增加 `requirePermission(permission)` 中间件。
- 接口按权限点拆分：`codes.write`、`products.write`、`orders.write`、`memberships.grant` 等。
- 后台 UI 按权限隐藏或禁用模块。
- 审计日志记录真实 admin id、IP、UA。

### 3.9 用户身份存在生产安全隐患

当前 `resolveUserId` 在 token 无效时，会继续接受：

- `x-demo-user-id`
- `query.userId`
- `body.userId`

影响：

- 如果生产环境没有严格关闭该路径，调用方可伪造任意用户身份。
- 购物车、订单、会员、下载权益均可能被越权访问或操作。

建议：

- 生产环境必须禁止 `x-demo-user-id`、`query.userId`、`body.userId` 身份回退。
- 除登录和公开内容外，所有 C 端用户接口都必须要求有效 Bearer Token。
- 小程序登录必须配置 `MINIAPP_SESSION_SECRET`、`WECHAT_APPID`、`WECHAT_APPSECRET`。

### 3.10 商城交易仍是演示交易，不是完整履约系统

当前已实现：

- 商品、SKU、价格、库存。
- 购物车、创建订单。
- mock 支付和微信支付 JSAPI 代码骨架。

主要缺口：

- 收货地址是硬编码演示地址。
- 没有用户地址管理。
- 没有物流公司、物流单号、发货时间。
- 没有退款接口和退款回调。
- 没有订单关闭/超时取消任务。
- 没有库存锁定，只有支付后扣库存。
- 没有真实微信支付配置验收。
- 没有交易类小程序订单发货管理对接。

官方参考：微信支付“小程序支付开发指引”要求小程序通过 JSAPI/小程序下单获取 `prepay_id` 后调起支付，并在支付后通过回调或查单确认状态；同时提示交易类小程序需要满足交易类小程序运营规范并接入订单发货管理功能，否则可能影响正式支付能力。

建议：

- 将商城标记为“演示完成，正式交易未完成”。
- 正式上线前补齐地址、物流、退款、关单、对账、发票入口、售后说明。

### 3.11 下载能力是本地演示，不是正式数字交付

当前实现：

- 服务端生成 15 分钟下载 ticket。
- `wx.downloadFile` 下载到临时文件。
- 服务端记录 IP、UA、下载次数。

主要缺口：

- 音频文件仍在本地目录，不是对象存储 + CDN。
- 没有真实文件 hash 校验。
- 没有断点续传或失败重试。
- 没有 `wx.saveFile` 或明确的小程序内离线保存方案。
- 没有 H5 下载页或 App 兜底。
- 音乐版权授权材料没有在项目中体现。

建议：

- 正式版采用对象存储私有桶 + CDN + 短时签名 URL。
- 下载 ticket 绑定用户、设备、IP 风险信息。
- 验收口径定义为“小程序内可控下载/离线缓存”，不要默认承诺可保存到手机公共文件系统。

### 3.12 缺少自动化测试和关键验收脚本

当前 `server/package.json` 没有 test/lint 脚本，仓库内未发现测试文件。

已执行：

```text
node --check server/miniprogram/tools 下 JS 文件：通过
createCodeBatch 函数级验证：失败，ReferenceError
```

建议至少补齐：

- 服务端单元测试：提取码、订单、支付、会员、下载。
- API 集成测试：完整 P0/P1/P2/P3 流程。
- 后台端到端测试：登录、生成批次、导出、改状态。
- 小程序手工验收清单或自动化截图验收。

## 6. 不合理或需要产品确认的点

1. “退出即结束”与“用户误退出容错”冲突  
   当前代码保留 30 分钟恢复能力。需要产品明确：到底要严格一次性，还是允许同一用户短时恢复。

2. 六位纯数字提取码安全性偏弱  
   100 万组合可被暴力尝试。需要 Redis 分布式限流、失败记录、设备/IP 风控。当前限流是内存级，重启或多实例会失效。

3. “一瓶酒 + 一首歌”与当前“一瓶酒 + 多首歌”冲突  
   需求是一个码绑定一首歌，但当前酒款默认有多个曲目。建议产品明确体验页是否只展示绑定曲目，其他曲目是否对会员开放。

4. 小程序内“下载歌曲”的验收边界要收紧  
   `wx.downloadFile` 下载的是临时文件路径，不等价于导出到手机公共文件系统。建议验收写成“小程序内可控下载/离线缓存”，如客户坚持外部播放器可用，需要 H5 或 App 兜底。

5. 会员/单曲虚拟商品支付合规需核验  
   数字音乐下载、会员权益、虚拟商品支付需要结合主体类目、微信支付权限、版权材料确认，不应默认与实物酒一样上线。

6. 酒类商城需要更完整合规  
   当前有提示文案，但正式上线还需要主体资质、商品资质、售后、隐私、支付、物流、未成年人保护和广告文案审查。

7. 后台“本地演示账号”不能出现在生产  
   `curator / Curator!2026` 只可用于开发环境。生产必须强制设置管理员初始密码、关闭默认账号提示。

## 7. 官方资料参考与对项目的影响

1. 微信支付“小程序支付开发指引”  
   链接：https://pay.wechatpay.cn/doc/v3/merchant/4012791911  
   对项目影响：正式交易应走“下单获取 prepay_id -> 小程序 requestPayment -> 回调/查单确认 -> 更新订单/发放权益”的闭环。当前实物订单有骨架，会员/单曲购买还没有按这个闭环实现。

2. 微信支付“JSAPI/小程序下单”  
   链接：https://pay.wechatpay.cn/doc/v3/merchant/4012791897  
   对项目影响：正式下单需要 appid、mchid、description、out_trade_no、notify_url、amount、payer.openid 等参数，并要确保商户订单号唯一。当前代码有这些字段基础，但真实配置和全流程验收未完成。

3. 微信支付“小程序调起支付”  
   链接：https://pay.wechatpay.cn/doc/v3/merchant/4012791898  
   对项目影响：前端要用服务端返回的调起支付参数调用 `wx.requestPayment`，并在支付返回后查单确认。当前实物订单页有该模式，会员和单曲解锁没有。

4. 微信支付“支付成功回调通知”  
   链接：https://pay.wechatpay.cn/doc/v3/merchant/4013070368  
   对项目影响：支付成功后的权益发放必须以后端回调或查单结果为准。当前虚拟权益直接由业务接口发放，不满足正式交易要求。

5. 微信开放文档“获取不限制的小程序码”  
   链接：https://developers.weixin.qq.com/miniprogram/dev/OpenApiDoc/qrcode-link/qr-code/getUnlimitedQRCode.html  
   对项目影响：固定入口码需要明确是微信公众平台生成还是服务端生成，并作为上线交付物保存。当前仅有未调用的生成函数。

6. 微信小程序 `wx.downloadFile` 文档  
   链接：https://developers.weixin.qq.com/miniprogram/dev/api/network/download/wx.downloadFile.html  
   对项目影响：小程序下载文件通常先获得临时路径。正式数字音乐下载需单独定义“保存/离线/外部可访问”的产品验收边界。
