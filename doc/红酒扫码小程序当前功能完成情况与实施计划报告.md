## 1. 结论摘要

当前项目已经不是早期的扫码音乐 MVP。结合代码实现看，项目已具备以下主能力：

1. 固定小程序入口 + 六位提取码 + 一次性消费 + 一码一酒一歌体验。
2. 前台商城、商品详情、购物车、订单确认、收货地址、JSAPI/mock 支付、库存预占、订单关闭、退款申请。
3. 后台登录、RBAC 权限、酒庄/酒款/曲目/提取码/商品/SKU/订单/会员/审计管理。
4. 会员购买、单曲解锁、下载签名、下载日志，且权益发放已改为支付成功后触发。
5. PostgreSQL 实体镜像、readiness/metrics、备份脚本、上线验收与运维手册、阶段 0-6 自动化测试。

因此，当前更准确的阶段判断是：

| 层级 | 当前判断 | 说明 |
|---|---|---|
| 本地演示完成度 | 高 | 主要业务域已能在开发环境闭环演示。 |
| 代码侧需求完成度 | 较高 | 阶段 0-6 的核心代码已落地，剩余重点是真实环境生产验收。 |
| 正式商用上线完成度 | 中等 | 真实微信登录、微信支付、交易类小程序发货管理、退款回调、对象存储/CDN、Redis、监控告警、合规材料仍需生产环境验收。 |

一句话结论：项目已达到“准生产内测版”的代码状态。本轮已补齐 P0/P1/P2 中可在代码侧完成的支付复用、音乐页解锁闭环、微信发货同步接口、微信退款接口/回调、CDN 下载跳转、Redis 限流适配和后台对账报表；但真实微信商户号、证书、域名、Redis、对象存储/CDN 与合规材料仍需在生产环境验收后，才能等同于“已正式商用上线”。

## 3. 当前代码实现总览

`miniprogram/app.json` 已注册 12 个页面：

|---|---|

| `pages/product-detail/index` | 商品详情、SKU、可售库存、加入购物车。 |

| `pages/order-confirm/index` | 收货地址录入、创建订单、JSAPI/mock 支付、支付状态轮询。 |
| `pages/member/index` | 会员状态、套餐购买、曲库、单曲解锁、下载记录。 |

小程序启动时会清理旧的提取码 session；`onHide` 也会清理体验态，符合“退出后体验结束”的产品方向。请求层优先使用用户 token，开发态缺 token 时才回退 demo user。

### 3.3 后台管理端

后台静态页位于 `server/public/admin/`。目前已具备：

1. 管理员登录、登出、改密码。
2. 按权限点隐藏菜单和操作按钮。
3. 酒款、酒庄、曲目、提取码批次、商品、SKU、订单、会员权益、审计日志管理。
4. 提取码批次支持按酒款筛选曲目，并提交 `trackId`。
5. 订单列表显示地址、物流、退款状态，并支持关闭超时订单。
6. 固定提取码小程序码生成入口。

### 3.4 数据与存储

当前数据层是“store snapshot + PostgreSQL 实体镜像”的混合模式：

1. `server/db/schema.sql` 保留 `app_state` JSONB 快照。
2. 同时存在 `wineries`、`wines`、`tracks`、`products`、`product_skus`、`users`、`user_addresses`、`orders`、`payments`、`refunds`、`memberships`、`download_entitlements`、`audit_logs` 等实体表。
3. `server/services/db-postgres-worker.js` 每次写入会同步实体镜像表，便于查询、报表和后续迁移。

这比纯 JSON 文件安全很多，但还不是完全表级 CRUD/事务模型。正式高并发上线前，提取码消费、库存预占、支付回调、权益发放、下载次数扣减仍建议进一步迁入数据库事务。

## 6. 当前最重要的缺口与风险

### 6.1 音乐页“解锁”按钮支付闭环已修复

本轮已新增 `miniprogram/utils/payment.js`，把订单确认页、会员中心、音乐页统一到同一套流程：

1. `/api/tracks/:trackId/unlock`
2. `/api/payments/orders/:orderId/jsapi`
3. `wx.requestPayment`
4. `/api/payments/orders/:orderId/status`
5. 刷新权益

现在 `pages/melody/index.js` 不再只创建待支付订单；只有 JSAPI/mock 支付成功并查单成功后，才提示已解锁并刷新曲目权益。用户取消支付时会提示“已取消支付”，不会误报解锁成功。

### 6.2 真实微信支付仍未验收

当前代码支持微信支付 JSAPI 下单、签名、回调验签和解密，也支持开发态 mock。但正式验收必须准备：

1. 小程序 AppID/AppSecret。
2. 微信支付商户号、商户证书/私钥、API v3 key。
3. 支付 notify_url HTTPS 域名。
4. 平台证书或微信支付公钥。
5. 真机 `wx.requestPayment` 测试。
6. 后端回调与查单补偿。

不能仅以 mock 支付通过作为正式完成。

### 6.3 交易类小程序订单发货管理代码已接入

代码内已有物流公司、物流单号、发货状态，本轮新增后台同步微信发货接口：

1. 后台订单可点击“同步微信发货”。
2. 服务端通过 `/api/admin/orders/:orderId/shipping/wechat` 组装 `upload_shipping_info` 请求。
3. 开发态无微信配置时走 mock 标记，生产态必须配置真实 AppID/AppSecret。
4. 同步成功、失败、失败原因都会写回订单并进入审计日志。

剩余风险是生产验收：必须用真实微信交易订单、真实物流信息和微信后台发货规范确认接口返回。

### 6.4 退款已具备微信接口与回调

当前用户可申请退款，后台可确认退款并按需入库。本轮新增：

1. `/api/admin/orders/:orderId/refund/wechat`：后台发起微信退款。
2. `/api/payments/wechat/refund-callback`：退款回调验签、解密和状态落库。
3. `/api/admin/refunds/:outRefundNo/refresh-wechat`：退款查单补偿。
4. `outRefundNo` 幂等字段和 `processing/refunded/failed` 状态。
5. 开发态无商户配置时可 mock 成功，生产态必须走真实商户接口。

### 6.5 数据库还不是完全事务化业务模型

当前 PostgreSQL 实体镜像适合报表和迁移过渡，但正式并发交易建议把以下动作改成数据库事务：

1. 提取码消费：使用 `where status='ready' returning *` 防并发重复消费。
2. 库存预占与释放：行级锁或事务更新。
3. 支付回调：按 `out_trade_no` 幂等更新订单、支付、权益。
4. 下载次数扣减：防止同一个权益并发超量下载。

### 6.6 媒体与下载已支持 CDN 交付模式

当前 `wx.downloadFile` 在本地开发态仍可下载服务端本地音频；本轮已加入生产交付模式：

1. `MEDIA_DELIVERY_MODE=cdn` 时，下载 ticket 被服务端消费后跳转到 `CDN_BASE_URL`。
2. 配置 `CDN_SIGNING_SECRET` 后会附带短时 `expires` 和 `signature`。
3. 下载仍先经过权益校验、ticket 校验、次数扣减和下载日志。
4. 生产还需迁移实际音频/封面/下载文件到对象存储，并校验 CDN 鉴权规则。

### 6.7 Redis、监控与对账已有代码基础

readiness 已把 Redis、对象存储/CDN 标为推荐项。本轮新增：

1. `server/services/redis-client.js`：可选 Redis 客户端。
2. 写操作、登录、提取码限流可使用 Redis 计数；无 Redis 时自动回退进程内限流。
3. `/api/admin/reports/reconciliation`：订单、支付、退款、发货、库存异常对账报告。
4. `/api/admin/orders/export`：订单 CSV 导出。
5. 后台订单页新增对账报告与订单导出入口。

生产仍需把 metrics、对账异常、支付/退款/下载失败日志接到外部监控和告警平台。

### 6.8 合规材料需要产品与主体确认

代码已有用户协议、隐私政策、理性饮酒提示，但正式提交审核前仍需确认：

1. 酒类经营主体资质和商品资质。
2. 未成年人保护与广告文案审查。
3. 微信小程序隐私接口声明。
4. 会员/数字音乐权益的支付类目与审核口径。
5. 音乐版权授权、下载授权、展示授权。
6. 售后、退款、物流、客服说明。
