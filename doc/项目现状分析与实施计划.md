# 项目现状分析与实施计划

- 分析日期：2026-04-16（阶段0、阶段1完成后更新）
- 分析基准：需求文档 v2.0 + 当前 main 分支代码
- 分析目标：逐项对照需求，明确已完成、未完成、需改造的模块，并给出详细实施计划

---

## 一、总体评估

**阶段0（阻塞修复与一码一歌模型）已完成。** 本次已修复提取码批次生成运行时错误，提取码批次和单码均支持绑定具体歌曲，消费提取码后只返回该码绑定的单首曲目，并新增阶段0/1测试脚本覆盖核心链路。

**阶段1（P0/P1 正式体验闭环）已完成。** 本次已关闭小程序首页的专属体验恢复入口，服务端会话恢复接口返回禁用状态；专属体验仅保存在本次小程序运行内存中，应用进入后台后清理体验数据。音乐页已改为展示真实曲目信息，并在专属入口尝试自动播放。后台新增固定提取码小程序码生成入口，配置真实 AppID/AppSecret 后可生成指向 `pages/redeem/index` 的固定码。

**阶段一（核心入口改造）已完成。** 项目已从 v1.0 的"一物一码扫码场景"模式全面改造为 v2.0 的"固定小程序码 + 六位提取码输入"模式。

**阶段二（安全与风控增强）已完成。** 提取码验证失败日志已实现，下载风控已增强为记录真实 IP 和设备信息。

**阶段三（内容正式化）已完成。** 后台已新增独立的酒庄管理模块和音乐资源管理模块，支持完整的增删改查操作。

**阶段四（合规与体验完善）已完成。** 用户协议、隐私政策、理性饮酒提示、未成年人保护提示均已实现，数据看板已增加提取码验证成功率、会员转化率、下单转化率、下载统计等深度指标。

**当前阶段0、阶段1已完成。** 剩余工作集中在上线环境部署、真实微信 AppID/AppSecret 与微信支付配置、管理员 RBAC、会员续费和生产验收。

## 三、阶段二改造详情

### 3.1 提取码验证失败日志 — 已完成 ✅

**改造前：**
- 验证失败时仅返回错误码，不记录任何日志
- 管理员无法查询谁在什么时候输错了什么码

**改造后：**
- 新增 `redeemFailLogs` 数据集合，记录每次验证失败
- 失败日志字段：`id`、`code`（输入的码）、`reason`（失败原因）、`ip`、`userId`、`createdAt`
- 失败原因枚举：`INVALID_FORMAT`、`CODE_NOT_FOUND`、`CODE_DISABLED`、`CODE_EXPIRED`、`CODE_ALREADY_USED`
- 每次失败同时写入审计日志（`code.redeem.failed`）
- 后台仪表盘第 4 张卡片改为"24h 验证失败"，展示最近 24 小时和历史的失败次数
- 新增 `GET /api/admin/redeem-fail-logs` 端点供后台查询

### 3.2 下载风控增强 — 已完成 ✅

**改造前：**
- 下载日志中 IP 固定为 `127.0.0.1`，设备信息固定为 `mini-program-demo`

**改造后：**
- `signDownload` 和 `consumeDownloadTicket` 接收 `requestMeta` 参数
- IP 取自请求的真实 IP（支持 `X-Forwarded-For` 头）
- 设备信息取自请求的 `User-Agent` 头
- 下载日志字段完整：`userId`、`trackId`、`assetId`、`ip`（真实 IP）、`deviceInfo`（真实 UA）、`downloadAt`
